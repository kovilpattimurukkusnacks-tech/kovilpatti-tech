# Performance Tips (Frontend TypeScript / React)

- **Memoize expensive derived data** with `useMemo` when it's recomputed from props/state that don't change every render — this repo already does this for category grouping in `ShopRequestNew.tsx` (`catGroups`, `leftCatGroups`/`rightCatGroups`).
- **`React.memo` + primitive props for hot list rows**: for large lists (100+ rows), memoize the row component and pass primitives instead of objects where possible, so only rows whose actual displayed value changed re-render. See `ProductRow` in `ShopRequestNew.tsx` for the pattern already established in this codebase — prefer matching it over introducing a new one.
- **Avoid heavy component libraries in hot render paths**: this repo deliberately uses a plain `<input>` instead of MUI `TextField` for the per-row qty input, because `TextField`'s nested styled-component tree was measured to be too slow multiplied across 200 rows. Don't "fix" this back to `TextField` without checking whether the row count that motivated it still applies.
- **`useDeferredValue`** for filter/search inputs that trigger expensive recomputation of a large list, so keystrokes stay responsive while the heavy recompute happens at lower priority (already used for `deferredProducts` in `ShopRequestNew.tsx`).
- **Don't run two independent async operations sequentially** when they don't depend on each other — use `Promise.all` / parallel React Query hooks instead of awaiting one after another.
- **Watch for effects that reset state on every render** because their dependency array includes a freshly-computed reference (e.g. an inline array/object literal, or `.map(...).join(...)` recomputed every render) — derive a stable primitive signature (like `catGroups.map(g => g.catId).join(',')`) if the goal is "changed contents", not "changed reference".
